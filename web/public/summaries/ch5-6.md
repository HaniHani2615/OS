## CHƯƠNG 5 — ĐỒNG BỘ HOÁ TIẾN TRÌNH (SYNCHRONIZATION)

### 5.1 Vấn đề: Bài toán Đoạn Tới Hạn (Critical Section)

- Khi nhiều tiến trình **cùng truy cập + cập nhật dữ liệu chia sẻ** → kết quả phụ thuộc vào thứ tự thực thi → **race condition**.
- **Đoạn tới hạn (CS)**: phần code thao tác lên biến chia sẻ.

**Ví dụ kinh điển — bộ đếm:**
```c
counter++;   // load → inc → store (3 lệnh, không atomic)
counter--;
```

### 5.2 ⭐ 3 yêu cầu của lời giải bài toán CS

1. **Mutual exclusion** (loại trừ lẫn nhau): chỉ 1 tiến trình ở CS tại 1 thời điểm.
2. **Progress** (tiến triển): nếu không ai trong CS, tiến trình muốn vào không bị **chặn vô hạn**; quyết định không trì hoãn vô hạn.
3. **Bounded waiting** (chờ có giới hạn): có giới hạn số lần tiến trình khác được vào CS sau khi tiến trình i đã yêu cầu.

### 5.3 Giải pháp Peterson (cho 2 tiến trình)

```c
// flag[i] = báo i muốn vào, turn = lượt ai
do {
    flag[i] = true;
    turn = j;
    while (flag[j] && turn == j) ; // chờ
    /* CRITICAL SECTION */
    flag[i] = false;
    /* REMAINDER */
} while (true);
```

⚠️ Giả sử các phép gán LOAD/STORE là atomic — **không đảm bảo trên kiến trúc hiện đại** (do reorder).

### 5.4 Phần cứng đồng bộ (atomic instructions)

#### `test_and_set`
```c
boolean test_and_set(boolean *t) {
    boolean rv = *t;
    *t = true;
    return rv;
}
```
Dùng: `while (test_and_set(&lock));`

#### `compare_and_swap` (CAS)
```c
int compare_and_swap(int *v, int exp, int new_val) {
    int rv = *v;
    if (*v == exp) *v = new_val;
    return rv;
}
```

### 5.5 ⭐ Mutex Lock

```c
acquire() {
    while (!available) ; // busy wait
    available = false;
}
release() { available = true; }
```
⚠️ **Spinlock** = busy wait → tốn CPU nhưng tránh context switch (tốt cho CS rất ngắn trên multi-core).

### 5.6 ⭐⭐ Semaphore (Dijkstra)

- Biến **nguyên** S, chỉ truy cập qua 2 phép atomic:
  - `wait(S)` = P: `while S<=0 do nothing; S--;`
  - `signal(S)` = V: `S++;`

**2 loại:**
- **Counting semaphore**: S ∈ ℤ
- **Binary semaphore** = mutex: S ∈ {0,1}

**Bảo toàn thứ tự**: P1 chạy `S1` rồi `signal(synch)`, P2 `wait(synch)` rồi chạy `S2` → đảm bảo S1 trước S2.

**Implementation tránh busy-wait:** kèm hàng chờ + `block()` / `wakeup()`.

### 5.7 ⭐ 3 bài toán đồng bộ kinh điển

#### a) Bounded Buffer (Producer–Consumer)
```
n   buffer slots
mutex (binary, 1)
empty (counting, n)   // số chỗ trống
full  (counting, 0)   // số chỗ đầy
```
- Producer: `wait(empty); wait(mutex); ... signal(mutex); signal(full);`
- Consumer: `wait(full); wait(mutex); ... signal(mutex); signal(empty);`

#### b) Readers–Writers
- Cho phép **nhiều readers đồng thời**, nhưng writer phải **độc quyền**.
- Biến: `read_count`, `mutex` (bảo vệ read_count), `rw_mutex` (writer/exclusive).
- ⚠️ **Starvation**: reader-priority có thể đói writer; writer-priority có thể đói reader.

#### c) Dining Philosophers (5 triết gia)
- 5 đũa = 5 semaphore. Mỗi triết gia: `wait(left); wait(right); ăn; signal(left); signal(right);`
- ⚠️ **Bế tắc** nếu mọi người cùng cầm đũa trái.
- Giải: cho phép tối đa 4 người cùng ăn / lấy cả 2 đũa cùng lúc / chẵn lấy trái trước, lẻ lấy phải trước.

### 5.8 Monitor (cấu trúc bậc cao)

- Đóng gói: dữ liệu chia sẻ + thủ tục thao tác → tự đảm bảo loại trừ lẫn nhau.
- **Condition variable**: `x.wait()`, `x.signal()`.
- `signal()` chỉ thức tỉnh **1** tiến trình đang chờ trên `x`.

### 5.9 ⚠️ LƯU Ý đồng bộ

1. **Mutex ≠ Semaphore ≠ Monitor** — tăng dần độ trừu tượng.
2. **Spinlock** chỉ tốt khi CS ngắn + CPU rảnh.
3. **`wait` giảm S, `signal` tăng S**.
4. Trật tự `wait(empty) → wait(mutex)` quan trọng — đảo ngược → **deadlock**.
5. Producer–Consumer: **không** dùng full/empty cho mutex bảo vệ buffer.

## CHƯƠNG 6 — BẾ TẮC (DEADLOCK)

### 6.1 Bài toán

Tập tiến trình mà mỗi tiến trình **giữ 1 tài nguyên** và **đợi 1 tài nguyên đang bị giữ bởi tiến trình khác trong tập**.

**Ví dụ:** Hai semaphore A=B=1
- P0: `wait(A); wait(B);`
- P1: `wait(B); wait(A);`

### 6.2 ⭐⭐ 4 ĐIỀU KIỆN CẦN CỦA BẾ TẮC (đồng thời)

1. **Mutual exclusion** — Loại trừ lẫn nhau: tài nguyên dùng độc quyền.
2. **Hold and wait** — Giữ và đợi: tiến trình giữ 1 tài nguyên + chờ thêm.
3. **No preemption** — Không dừng: tài nguyên chỉ giải phóng tự nguyện.
4. **Circular wait** — Chờ vòng tròn: chuỗi P0→P1→…→Pn→P0.

### 6.3 Đồ thị phân phối tài nguyên (Resource Allocation Graph)

- Đỉnh: P (tiến trình), R (tài nguyên — số dấu chấm = số đơn vị).
- Cạnh:
  - **Yêu cầu**: `Pi → Rj`
  - **Cấp phát**: `Rj → Pi`
- ⚠️ **Có chu trình + mỗi tài nguyên 1 đơn vị → CHẮC CHẮN bế tắc.**
- Có chu trình + nhiều đơn vị/tài nguyên → **có thể** bế tắc, **có thể** không.

### 6.4 ⭐ 4 phương pháp xử lý bế tắc

| Phương pháp | Cách làm |
|---|---|
| **Phòng tránh** (Prevention) | Loại bỏ 1 trong 4 điều kiện |
| **Tránh** (Avoidance) | Dùng thông tin trước → kiểm tra trạng thái an toàn |
| **Phát hiện & khôi phục** | Cho phép xảy ra, phát hiện rồi khôi phục |
| **Bỏ qua** | Linux, Windows mặc định chọn cách này |

### 6.5 Phòng tránh — loại bỏ điều kiện

- **Mutual exclusion**: dùng tài nguyên chia sẻ (nếu được).
- **Hold and wait**: yêu cầu cấp phát **toàn bộ** trước hoặc chỉ yêu cầu khi **không giữ gì**. ⚠️ Mức sử dụng thấp + nạn đói.
- **No preemption**: nếu yêu cầu không thoả → **giải phóng tất cả** đang giữ.
- **Circular wait**: **đánh số tài nguyên**, yêu cầu theo thứ tự **tăng dần**.

### 6.6 ⭐ Trạng thái an toàn (Safe state)

Có chuỗi `<P1, P2, …, Pn>` sao cho với mỗi Pi: tài nguyên Pi cần thêm ≤ tài nguyên có sẵn + tài nguyên các Pj (j<i) đang giữ.

**Quan hệ:**
```
Safe ⊊ Không bế tắc.   Unsafe ⊋ có thể (chưa chắc) bế tắc.
```

### 6.7 ⭐⭐ Thuật toán Banker (Avoidance)

**Cấu trúc dữ liệu (n tiến trình, m loại tài nguyên):**

| Tên | Mô tả |
|---|---|
| `Available[m]` | số đơn vị mỗi loại đang sẵn có |
| `Max[n][m]` | tối đa Pi có thể yêu cầu |
| `Allocation[n][m]` | đang cấp phát cho Pi |
| `Need[n][m]` | = `Max - Allocation` |

**Thuật toán an toàn (Safety):**
```
1. Work := Available, Finish[i] := false ∀i
2. Tìm i: Finish[i]==false ∧ Need[i] ≤ Work
   nếu không có → 4
3. Work := Work + Allocation[i], Finish[i] := true → 2
4. Nếu Finish[i]==true ∀i → AN TOÀN
```

**Thuật toán yêu cầu (Request):** với `Request_i`:
```
1. Nếu Request_i > Need_i → LỖI (vượt khai báo)
2. Nếu Request_i > Available → Pi phải chờ
3. Giả định cấp phát:
   Available -= Request_i
   Allocation_i += Request_i
   Need_i -= Request_i
   Chạy thuật toán an toàn:
     - An toàn → cấp phát thật
     - Không → khôi phục, Pi chờ
```

**Ví dụ:** 5 tiến trình, A=10/B=5/C=7. Trạng thái t0:

| | Alloc(ABC) | Max(ABC) | Need(ABC) |
|---|---|---|---|
| P0 | 010 | 753 | 743 |
| P1 | 200 | 322 | 122 |
| P2 | 302 | 902 | 600 |
| P3 | 211 | 222 | 011 |
| P4 | 002 | 433 | 431 |

Available = (3,3,2). Chuỗi an toàn: **<P1, P3, P4, P2, P0>**.

### 6.8 Phát hiện bế tắc (Detection)

#### Một đơn vị / tài nguyên → **Wait-for graph**
- Đỉnh: tiến trình. Cạnh `Pi → Pj` ⟺ Pi chờ Pj.
- **Có chu trình ⟺ bế tắc**.

#### Nhiều đơn vị → biến thể Banker
Tương tự safety nhưng dùng `Request` thay `Need`. Độ phức tạp $O(m \cdot n^2)$.

### 6.9 Khôi phục bế tắc

- **Dừng tiến trình**: tất cả các tiến trình bế tắc / từng tiến trình một.
  - Tiêu chí: ưu tiên, thời gian đã chạy, tài nguyên đã dùng/còn cần, tương tác hay batch.
- **Dừng tài nguyên** (preempt): chọn tài nguyên + tiến trình bị dừng → **rollback** đến trạng thái an toàn → khởi động lại.
- ⚠️ **Nạn đói**: 1 tiến trình luôn bị chọn dừng.

## ⚠️ LƯU Ý ÔN TẬP

1. **Critical section** không phải là vùng nhớ — là **đoạn code**.
2. Peterson chỉ đúng cho **2 tiến trình**.
3. **Semaphore khởi tạo** tuỳ ý nghĩa: mutex=1, full=0, empty=n.
4. **Wait/signal phải atomic** — đó là nhiệm vụ của HĐH.
5. **Banker không tối ưu** — từ chối nhiều yêu cầu hợp lệ để tránh trạng thái unsafe.
6. **4 điều kiện bế tắc**: nhớ thứ tự **M-H-N-C** (Mutex / Hold-wait / No-preempt / Circular).
7. **Nạn đói (starvation) ≠ bế tắc (deadlock)**: bế tắc → tất cả dừng, nạn đói → có tiến trình chạy nhưng 1 tiến trình mãi không tới lượt.
8. **Wait-for graph chỉ dùng được khi mỗi tài nguyên có 1 đơn vị**.
9. Bài tập hay gặp: cho ma trận → tìm chuỗi an toàn → nếu Pi yêu cầu (a,b,c) có cấp phát được không?
